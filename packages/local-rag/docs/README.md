## Documentation Philosophy

### Principles

1. **User-first**: Write for the reader, not the writer
2. **Examples**: Show, don't just tell
3. **Searchable**: Use clear headings and keywords
4. **Up-to-date**: Keep docs in sync with code
5. **Accessible**: Write clearly, avoid jargon

### Structure

Each document follows this pattern:

```markdown
# Title

Brief introduction (what and why)

---

## Sections with clear headings

Content organized logically

### Sub-sections as needed

Examples and code blocks

---

## References

Links to related docs
```

## Contributing to Documentation

### When to Update Docs

- **Before coding**: Design docs, architecture decisions
- **During coding**: Code comments, docstrings
- **After coding**: User guides, changelog
- **Always**: Keep docs accurate and current

### How to Contribute

1. **Find outdated info**: Open an issue
2. **Propose improvements**: Open a discussion
3. **Submit changes**: Create a pull request
4. **Add examples**: Always welcome!

### Style Guide

- Use Markdown with GitHub flavor
- Keep lines < 100 characters (for diffs)
- Use code blocks with language tags
- Include examples for complex concepts
- Link between related documents

### New Feature Documentation

```markdown

Brief description

Technical details

Code or usage examples

Any settings
```

### Bug Report

```markdown
# Bug: Brief Description

## Description
What's wrong

## Reproduction
Steps to reproduce

## Expected vs Actual
What should happen vs what does

## Workaround
If known

## Fix Plan
How to fix (optional)
```
